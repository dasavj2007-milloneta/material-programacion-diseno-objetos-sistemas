Proyecto de Diagramación de Clases y Desarrollo OO con SOLID

Cada grupo diseñará el diagrama de clases UML de un sistema real y lo implementará en Java aplicando los cinco principios SOLID. El proyecto integra modelado, diseño y buenas prácticas de ingeniería de software.

1. Objetivos del Proyecto

📐 Modelado UML

Diseñar un diagrama de clases completo con atributos, métodos, visibilidad, relaciones y multiplicidad correctamente representados.

🏗️ Principios SOLID

Aplicar los cinco principios SOLID en el diseño e implementación del sistema, justificando cada decisión de diseño.

💻 Implementación OO

Traducir el diagrama de clases a código funcional aplicando herencia, polimorfismo, composición, interfaces y encapsulamiento.

📄 Documentación

Crear documentación técnica que relacione cada principio SOLID con decisiones concretas del diseño y la implementación.

2. Principios SOLID a Aplicar

Cada proyecto debe demostrar los cinco principios

🔴 S — Single Responsibility

Cada clase debe tener una única razón para cambiar.

Separar responsabilidades en clases distintas. Ejemplo: no mezclar lógica de negocio con persistencia de datos.

🟠 O — Open/Closed

Las entidades deben estar abiertas para extensión pero cerradas para modificación.

Usar herencia o interfaces para agregar funcionalidad sin modificar el código existente.

🟡 L — Liskov Substitution

Los objetos de una subclase deben poder sustituir a los de la superclase sin alterar el comportamiento.

Las clases hijas deben respetar el contrato de la clase padre.

🟢 I — Interface Segregation

Es mejor tener interfaces pequeñas y específicas que una interfaz grande y general.

Dividir interfaces grandes en varias más cohesivas.

🔵 D — Dependency Inversion

Depender de abstracciones, no de implementaciones concretas.

Usar interfaces o clases abstractas para desacoplar módulos de alto y bajo nivel.

📌 Requisito Obligatorio

En la documentación del proyecto, cada grupo debe incluir una sección titulada "Aplicación de Principios SOLID" donde se explique, con fragmentos de código y referencia al diagrama UML, cómo se aplicó cada principio.

3. Entregables del Proyecto

📊 Entregable 1 — Diagrama de Clases UML

  • Mínimo 6 clases con atributos, métodos y visibilidad
  • Al menos 3 tipos de relación (herencia, composición, agregación, asociación o dependencia)
  • Multiplicidad correctamente indicada en todas las relaciones
  • Al menos 1 clase abstracta y 2 interfaces
  • Herramienta: draw.io, Lucidchart, StarUML o similar

💻 Entregable 2 — Implementación en Código

  • Implementación en Java
  • Código que refleje fielmente el diagrama de clases
  • Aplicación demostrable de los 5 principios SOLID
  • Clase Main que demuestre el funcionamiento
  • Repositorio en GitHub con commits descriptivos

📝 Entregable 3 — Documentación Técnica

  • Descripción del sistema y su contexto
  • Diagrama de clases UML exportado en imagen
  • Explicación de cada relación y su justificación
  • Sección "Aplicación de Principios SOLID" con código de ejemplo
  • Conclusiones y lecciones aprendidas

⚠️ Formato de Entrega

  • Repositorio GitHub: Nombre del repo [nombre-proyecto]-grupo-[N]
  • README.md: Debe contener la documentación técnica completa
  • Carpeta /diagrams: Imágenes del diagrama UML (.png o .svg)
  • Carpeta /src: Código fuente organizado por paquetes

4. Rúbrica de Evaluación

Criterio Excelente (5.0) Bueno (4.0) Aceptable (3.0) Deficiente (< 3.0) Peso
Diagrama de Clases UML ≥ 6 clases, relaciones correctas, multiplicidad, clase abstracta e interfaces. Notación UML impecable. 5-6 clases, mayoría de relaciones correctas, pequeños errores de notación. 4-5 clases, relaciones incompletas, errores de multiplicidad. Menos de 4 clases, relaciones incorrectas o ausentes. 25%
Principios SOLID Los 5 principios aplicados y justificados con código. Diseño evidencia comprensión profunda. 4 principios aplicados correctamente, justificación parcial. 3 principios aplicados, justificación superficial. Menos de 3 principios o aplicación incorrecta. 30%
Implementación OO Código funcional, bien estructurado, herencia/polimorfismo/interfaces correctos. Compilación sin errores. Código funcional con pequeñas inconsistencias frente al diagrama. Código parcialmente funcional, falta implementar algunas clases. Código no compila o no refleja el diagrama. 25%
Documentación README completo, diagramas exportados, relación SOLID-código clara, conclusiones reflexivas. Documentación completa pero sin profundidad en SOLID. Documentación parcial, falta sección SOLID o diagramas. Documentación ausente o incompleta. 10%
Sustentación Ambos integrantes explican con claridad el diseño, código y principios aplicados. Se explica correctamente pero con dudas puntuales. Solo un integrante puede explicar el proyecto. No se puede sustentar el proyecto. 10%

🚨 Nota Importante

Proyectos con código copiado de internet sin adaptación ni comprensión demostrable recibirán nota de 0.0. Se verificará la originalidad del diseño y la capacidad de sustentación de cada integrante.

  • Hacer commits frecuentes con mensajes descriptivos
